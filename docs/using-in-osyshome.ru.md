# Привязка к osysHome: `linked_object` и свойства

Плагин `Mqtt` хранит настройки топиков в своей БД (`mqtt_topics`) и использует привязки к объектам osysHome. Пользователь настраивает привязку в интерфейсе `Mqtt -> Add topic` полями:

1) `Linked object` (имя объекта osysHome)
2) `Linked property` (имя свойства объекта)
3) `Linked method` (имя метода объекта)

В зависимости от заполненных полей плагин делает следующее.

## Входящие сообщения (MQTT -> osysHome)

Плагин обрабатывает входящее MQTT сообщение, когда:
1) оно пришло на подписанный брокером топик (из `Mqtt -> Settings -> Subscribe topics`),
2) и `msg.topic` точно совпадает с `Topic.path` в таблице.

Дальше для каждой записи `Topic`:

1) Значение декодируется как UTF-8 строка (или фиксируется как “Binary data…” при ошибке декодирования).
2) Если задан `Replace list`, значение преобразуется перед записью в осysHome.
3) Если задан `Linked method`, плагин вызывает `Объект.Метод`.
4) Если задан `Linked property`, плагин обновляет `Объект.Свойство`.

Важно: для входящих команд есть фильтр `"/set"` в топике. Если в `msg.topic` встречается подстрока `"/set"`, сообщение игнорируется.

## Исходящие сообщения (osysHome -> MQTT)

Когда в osysHome изменяется свойство объекта (например, вы выполнили `setProperty("Obj.prop", value)`), ObjectManager вызывает у подключённых модулей метод `changeLinkedProperty(...)`.

Плагин `Mqtt` находит все топики, где:
- `mqtt_topics.linked_object = Объект`
- `mqtt_topics.linked_property = prop`

и публикует новое значение в MQTT:
- в `Topic.path`, либо в `Topic.path_write`, если оно задано

Ограничение: если `Readonly = true` у топика, то исходящие публикации не выполняются.

## Какие свойства обновляются и какие типы поддерживаются

Плагин не “знает” заранее конкретные имена свойств: он обновляет именно то свойство, которое вы указали в `Linked property`.

При обновлении используется механизм osysHome для типов свойства. Поддерживаются типы (по логике конвертации в osysHome):
- `int` — строка -> `int`, возможны проверки `min/max/step/allowed_values`
- `float` — строка -> `float`, возможны проверки `min/max/step/decimals/allowed_values`
- `str` — строка
- `datetime` — строка парсится как datetime
- `dict` — JSON строка -> dict
- `list` — JSON строка -> list
- `bool` — принимаются значения (case-insensitive): `true/1/t/y/yes/on` и `false/0/f/n/no/off`
- `enum` — проверяется по `enum_values` (если задано)

Важно: если свойство ещё не существует в объекте, osysHome создаст его автоматически, но с типом `str` (потому что `Mqtt` передаёт в `setProperty/updateProperty` именно строку).
Чтобы JSON приходил в `dict/list` корректно, заранее создайте свойство нужного типа (`dict` или `list`).

## Как работает `Replace list`

`Replace list` задаётся строкой формата:
`FROM=TO,FROM2=TO2`

1) MQTT -> osysHome: используется сопоставление “входное значение FROM -> значение TO”.
2) osysHome -> MQTT: используется обратное сопоставление “значение TO -> FROM”.

Пример: `OPEN=1,CLOSED=0`
- вход `OPEN` -> свойство получает `1` (затем конвертация в `bool`/`int` зависит от типа свойства)
- выход `True/1` -> публикуется `OPEN`

Важно: замены ищутся по точному совпадению строки (пробелы в `Replace list` не тримятся).

## `Linked method`: какие аргументы получит метод

Если задан `Linked method`, на входящие сообщения выполняется вызов `callMethodThread(...)`.

В код метода (в осysHome) передаётся переменная `params` (она содержит переданные аргументы). В частности, для обычной (UTF-8) декодировки плагин передаёт:
- `params.VALUE` — текущее значение из MQTT
- `params.NEW_VALUE` — то же значение (как для “нового”)
- `params.OLD_VALUE` — предыдущее значение топика (из таблицы `mqtt_topics`)
- `params.TITLE` — `Topic.title`

## Быстрая проверка в автоматизациях

Чтение значения:

```python
val = getProperty("MqttWeather.payload")
logger.info("payload = %s", val)
```

Запись значения (и проверка публикации в MQTT):

```python
setProperty("MQTTRelay.state", True)
```

Если настроено `path_write` и `Readonly = false`, то это приведёт к публикации в MQTT.
